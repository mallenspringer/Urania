# Urania Implementation Sequence Roadmap

## Purpose

This document outlines the recommended, dependency-ordered sequence for developing the Urania application. Following this roadmap ensures that foundational data models and computational layers are verified before building rendering components, interaction tools, and user interface panels.

---

## Phases at a Glance

```text
Phase 1: Foundations & Core Model
    ↓
Phase 2: Command & Undo Infrastructure
    ↓
Phase 3: Mechanism Engine & Resolvers
    ↓
Phase 4: Rendering Surface & Dumb Renderers
    ↓
Phase 5: Selection & Canvas Interaction
    ↓
Phase 6: Tool Registry & Creation Tools
    ↓
Phase 7: Precision Property System
    ↓
Phase 8: Validation & Verification Layer
    ↓
Phase 9: Export Pipeline & Interactive Runtimes
    ↓
Phase 10: Templates & UI Polish
```

---

## Phase 1: Foundations & Core Model

Build the mathematical and logical structures upon which all other features rely.

### Tasks
1. **Math Utilities**: Implement pure polar coordinate utilities (Cartesian-to-Polar conversion, angle normalization between $0^\circ$ and $360^\circ$, clockwise positive rotation vectors).
2. **TypeScript Definitions**: Define strict interfaces for the Scene Graph nodes (`Project`, `Ring`, `Sector`, `Element`, `Transform`, `Metadata`, `Settings`) in `src/shared/types/`.
3. **Zustand Project Store**: Set up the central state store (`ProjectStore`) containing the project schema, version, active ring metadata, and initial empty project bootstrap state.
4. **Serialization & JSON Helpers**: Write deterministic serialize/deserialize methods to load and save the project JSON model, ensuring unknown properties are preserved verbatim.

---

## Phase 2: Command & Undo Infrastructure

Establish the exclusive mutation pathway before writing any state-changing code.

### Tasks
1. **Command Interface**: Define the abstract `Command` structure with `execute()`, `undo()`, and `getLabel()` hooks.
2. **Command History Manager**: Build the history stack with undo/redo pointer limits (default 100 entries).
3. **Initial Structural Commands**: Write commands for basic scene manipulation:
   * `CreateRingCommand`
   * `DeleteRingCommand`
   * `RotateRingCommand`

---

## Phase 3: Mechanism Engine & Resolvers

Implement the computational layer that answers how physical elements behave and transform.

### Tasks
1. **Transform Resolver**: Implement recursive parent-to-child matrix transformation resolvers to determine absolute coordinates from relative properties.
2. **Volvelle Adapter**: Write the volvelle coordination rules:
   * Evaluate ring rotation layers.
   * Expand procedural pattern instructions into temporary instances.
   * Compute pixel-accurate clipping regions and reveal intersections for windows.
3. **Engine Output Interface**: Expose clean data hooks presenting fully resolved nodes (`ResolvedNode` lists with world-space coordinates and computed visibilities).

---

## Phase 4: Rendering Surface & Dumb Renderers

Set up the visual workspace utilizing Canvas/Konva.js.

### Tasks
1. **Viewport System**: Initialize the HTML5 Canvas wrapper, viewport pan, pan limits, and cursor-centered logarithmic zoom.
2. **Rings & Sector Renderers**: Build presentation renderers that draw concentric ring bands and wedge-based sectors.
3. **Element Renderers**: Implement rendering for shapes (rectangles, circles, polygons) and text (standard, arc-text layout, sector labels).
4. **Dynamic Masking Implementation**: Apply computed reveal masks using Konva’s masking options, feeding them standard SVG path strings generated by the engine.

---

## Phase 5: Selection & Canvas Interaction

Develop interaction handlers and link them directly to the selection store.

### Tasks
1. **Selection Store**: Initialize selections/hover stores containing explicit selection models (`RingSelection`, `ObjectSelection`, etc.).
2. **Direct Hit-Testing Bias**: Implement canvas click listeners honoring lock states, hidden visibility, and active-ring priority bias.
3. **Canvas Selection Highlights**: Draw selection outlines, bounding boxes, and active rotation handles.
4. **Marquee Selector**: Implement drag-to-select marquee box functionality with automatic touch checks.

---

## Phase 6: Tool Registry & Creation Tools

Provide tools to create and alter elements.

### Tasks
1. **Tool Registry**: Implement the centralized `ToolRegistry` to coordinate active tools, cursors, and capability flags.
2. **Select Tool**: Wire the default mouse cursor tool to handle moving, resizing, and dragging elements.
3. **Creation Tools**: Develop tool controllers executing the standard `Activate` $\rightarrow$ `Preview` $\rightarrow$ `Validate` $\rightarrow$ `Command` lifecycle for:
   * Shapes (rectangles, circles, polygons)
   * Windows
   * Arc Text & Labels
   * Concentric/Radial Guides

---

## Phase 7: Precision Property System

Add property sheets for granular editing.

### Tasks
1. **Inspector Panel Layout**: Build the inspector container in the right sidebar.
2. **Object Specific Panels**: Write property sheets showing configurations for selected elements:
   * Dimensions, radii, angles, text contents, font styles, and path values.
3. **Continuous Updates**: Wire numeric input sliders and scrubbers to dispatch transactional commands on interact change.

---

## Phase 8: Validation & Verification Layer

Introduce structural checks and warnings.

### Tasks
1. **Validator Framework**: Build continuous validation hooks executing on command commits.
2. **Geometric Constraints**: Implement validations verifying positive radii, valid spans, and asset integrity.
3. **Validation Panel UI**: Create a status dashboard summarizing errors, warnings, and fabrication alerts (e.g., cutout bridges too thin for physical paper cutting).

---

## Phase 9: Export Pipeline & Interactive Runtimes

Implement file outputs.

### Tasks
1. **SVG Exporter**: Compile vector exports (artwork, cut paths, and fold paths) mapping resolved node geometries to separate files or grouped layers.
2. **Interactive HTML Generator**: Package the SVG runtime rendering container, styling rules, local storage state persistence, and mechanism engine scripts into a self-contained ZIP bundle.
3. **Export Dashboard UI**: Build UI configuration controls to preview exports and triggers.

---

## Phase 10: Templates & UI Polish

Tie the application workflow together.

### Tasks
1. **Template Manager**: Implement parsing logic to initialize new project files using template profiles.
2. **Home Screen / Dashboard**: Create the startup splash screen showing template card lists, instructions, and file upload dropzones.
3. **Autosave & Preferences**: Wire command-counter-based local storage backups and custom preferences.
